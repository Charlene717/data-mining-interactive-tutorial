資料探勘導論與 CRISP-DM

從定義、流程到 2025 年的演進——掌握資料探勘專案的全景方法論。

From definitions to process to the 2025 landscape — master the panoramic methodology of every data mining project.

概覽

什麼是資料探勘？

資料探勘 (Data Mining) 是從大量資料中挖掘出非顯而易見、潛在有用、可解釋的模式 (patterns) 與知識 (knowledge) 的過程。它座落於統計學、機器學習、資料庫與領域知識的交集，是 KDD (Knowledge Discovery in Databases) 流程中的核心步驟。

在生物醫學脈絡下，資料探勘的對象橫跨 omics 矩陣、EHR 結構化欄位、影像、PubMed 文獻、臨床路徑等異質資料，常見任務包含：分類疾病 vs 健康、分群病人亞型、發現基因—疾病關聯、預測再入院風險。

Data Mining is the process of extracting non-obvious, potentially useful, interpretable patterns and knowledge from large datasets. It sits at the intersection of statistics, machine learning, databases, and domain expertise — and is the core step inside KDD (Knowledge Discovery in Databases).

In a biomedical context, data mining targets omics matrices, structured EHR fields, images, PubMed literature, and clinical pathways. Typical tasks include disease vs healthy classification, patient subgroup clustering, gene–disease association discovery, and readmission risk prediction.

💡
核心原則：資料探勘不是「跑模型就好」。每一個成功的專案——尤其是醫療領域——必定包含明確的業務問題、嚴謹的資料理解、與審慎的部署評估。方法論（不是演算法）才是專案成敗的關鍵。 Core principle: Data mining is never "just run a model." Every successful project — especially in healthcare — must contain a well-framed business question, rigorous data understanding, and careful deployment evaluation. Methodology, not the algorithm, decides project success.
任務分類

一、資料探勘的六大核心任務

🎯

分類 Classification

給定已標記資料，學習把新樣本歸入離散類別。例：判斷腫瘤良/惡性。代表方法 Logistic、Tree、SVM、NN。

Learn from labeled data to assign new samples into discrete classes. e.g. benign vs malignant tumor. Methods: Logistic, Tree, SVM, NN.

📈

迴歸 Regression

預測連續數值。例：估算病患住院天數、血糖值。代表方法線性、Ridge、隨機森林迴歸、XGBoost。

Predict a continuous numeric value. e.g. length of stay, blood glucose. Methods: linear, Ridge, random forest regression, XGBoost.

🧩

分群 Clustering

非監督學習：發現資料中的自然群組。例：病人亞型發現。代表方法 K-Means、Hierarchical、DBSCAN、GMM。

Unsupervised: find natural groups in data. e.g. patient subtype discovery. Methods: K-Means, Hierarchical, DBSCAN, GMM.

🛒

關聯規則

從交易/共現中找「A → B」規則。例：購物籃分析、藥物—副作用共現、症狀群組。Apriori、FP-Growth、Eclat。

Find "A → B" rules from transactional/co-occurrence data. e.g. market basket, drug–adverse event co-occurrence. Apriori, FP-Growth, Eclat.

🚨

異常偵測 Anomaly

標識偏離常態的觀測值。例：信用卡詐欺、ICU 生理異常、實驗批次效應。Isolation Forest、LOF、ECOD、Autoencoder。

Identify observations that deviate from the norm. e.g. credit-card fraud, ICU physiologic anomaly, batch effects. Isolation Forest, LOF, ECOD, Autoencoder.

🔗

序列樣式 Sequence

從有時間或順序的資料中挖掘共同模式。例：臨床用藥路徑、網站點擊流。GSP、SPADE、PrefixSpan。

Find common patterns from time- or order-ordered data. e.g. clinical prescription pathways, clickstreams. GSP, SPADE, PrefixSpan.

CRISP-DM

二、CRISP-DM 六階段流程

CRISP-DM (Cross-Industry Standard Process for Data Mining) 由 1996 年 SPSS、Daimler-Benz、NCR 共同提出，至 2025 年仍是業界最廣泛使用的方法論。即使後續出現 KDD、SEMMA、TDSP、bizML、CRISP-ML(Q) 等變體，多數仍保留它的六階段骨架——只是擴充了 MLOps / 模型治理 / AI 倫理等步驟。

CRISP-DM (Cross-Industry Standard Process for Data Mining) was proposed in 1996 by SPSS, Daimler-Benz, and NCR. As of 2025 it remains the most widely adopted methodology in industry. Even newer variants — KDD, SEMMA, TDSP, bizML, CRISP-ML(Q) — preserve its six-phase skeleton and only extend it with MLOps / model governance / AI ethics steps.

業務理解

釐清目標與限制：你到底要回答什麼問題？成功標準是什麼？例：「為 2026 年 Q3 上線的 30 天再入院預測模型定義 AUROC ≥ 0.78 並可解釋。」

Clarify goal and constraints: what question are you answering? Success criteria? e.g. "AUROC ≥ 0.78, interpretable, deployed Q3 2026 for 30-day readmission."

資料理解

收集、描述、探索、驗證資料品質。常見產出：data dictionary、缺失率報表、單變量分佈圖。

Collect, describe, explore, and verify data quality. Common outputs: data dictionary, missingness report, univariate distribution plots.

資料準備

清理、整合、變形、特徵工程。實務上佔總時間 60–80%。產出：可分析的 analytic dataset。

Cleaning, integration, transformation, feature engineering. In practice 60–80% of total time. Output: ready-to-model analytic dataset.

模型建構

選擇與訓練模型、調參、做 cross-validation。注意：常需回到第 3 步補做特徵工程。

Select and train models, tune hyperparameters, run cross-validation. Note: you frequently loop back to Step 3 for more feature engineering.

評估

不只看 AUROC——還要評估業務影響、fairness、可解釋性。是否真能回答 Step 1 的問題？

Don't just look at AUROC — also evaluate business impact, fairness, interpretability. Does it actually answer the Step 1 question?

部署

整合到正式系統、文件化、監控漂移 (drift)、定期再訓練。這是 CRISP-DM 最常被輕忽的階段。

Integrate into production, document, monitor drift, schedule retraining. This is the most under-invested phase of CRISP-DM.

⚠️
CRISP-DM 的特色是「循環」而非「線性」：箭頭從 Evaluation 可以回到 Business Understanding；從 Modeling 可以回到 Data Preparation。實務上一個 EHR 探勘專案往往循環 3–5 次。 CRISP-DM is "cyclic," not linear: arrows can flow from Evaluation back to Business Understanding, and from Modeling back to Data Preparation. A real-world EHR mining project typically iterates 3–5 times.

🌳 該選哪個方法論？

Q1:
學術或概念性練習？→ 是 → 使用 KDD（含 Data Selection / Preprocessing / Transformation / Mining / Interpretation）。
Q2:
實務專案、需向業務說明？→ 是 → 用 CRISP-DM；六階段語言業界都熟。
Q3:
專案最終要部署 MLOps？→ 是 → 用 CRISP-ML(Q) 或 TDSP，補上 QA / Monitoring / Retraining 環節。
Q4:
專案要回答業務 KPI 而非演算法精度？→ 是 → 採 bizML，用「業務語言」對齊團隊。
Q1:
Academic or conceptual exercise? → Yes → Use KDD (Data Selection / Preprocessing / Transformation / Mining / Interpretation).
Q2:
Real project, must explain to business? → Yes → Use CRISP-DM — the six-phase vocabulary is universal in industry.
Q3:
Project ends in production MLOps? → Yes → Use CRISP-ML(Q) or TDSP for QA / Monitoring / Retraining.
Q4:
Project must answer business KPI, not algorithmic accuracy? → Yes → Use bizML for business-language alignment.
程式碼

實作骨架：CRISP-DM 在 R / Python

# 1. Business understanding → 寫成註解 / Write as comments
# Goal: predict 30-day hospital readmission. Target AUROC >= 0.78

# 2. Data understanding
library(tidyverse); library(tidymodels); library(skimr)
df <- read_csv("readmission.csv")
skim(df)             # 缺失、分佈、唯一值 / missing, distribution, uniqueness

# 3. Data preparation — recipe-based workflow
rec <- recipe(readmit ~ ., data = df) |>
  step_impute_median(all_numeric_predictors()) |>
  step_dummy(all_nominal_predictors()) |>
  step_normalize(all_numeric_predictors())

# 4. Modeling
mod <- logistic_reg(penalty=tune()) |> set_engine("glmnet")
wf  <- workflow() |> add_recipe(rec) |> add_model(mod)
folds <- vfold_cv(df, v=10, strata=readmit)

# 5. Evaluation
res <- tune_grid(wf, folds, metrics=metric_set(roc_auc, pr_auc))
show_best(res, "roc_auc")

# 6. Deployment — package final model + monitoring hook
library(vetiver)
v <- vetiver_model(fit(finalize_workflow(wf, select_best(res)), df),
                  "readmit_model_v1")
import pandas as pd
from sklearn.model_selection import StratifiedKFold, GridSearchCV
from sklearn.pipeline       import Pipeline
from sklearn.compose        import ColumnTransformer
from sklearn.preprocessing  import StandardScaler, OneHotEncoder
from sklearn.impute         import SimpleImputer
from sklearn.linear_model   import LogisticRegression
from sklearn.metrics        import roc_auc_score

# 1+2 Business + Data understanding
df = pd.read_csv("readmission.csv")
y  = df.pop("readmit")
df.describe(include="all"); df.isna().mean()

# 3. Data preparation (column transformer)
num = df.select_dtypes("number").columns
cat = df.select_dtypes("object").columns
pre = ColumnTransformer([
    ("num", Pipeline([("imp", SimpleImputer(strategy="median")),
                       ("sc",  StandardScaler())]), num),
    ("cat", OneHotEncoder(handle_unknown="ignore"), cat)
])

# 4. Modeling + CV
pipe = Pipeline([("pre", pre), ("clf", LogisticRegression(max_iter=2000))])
cv   = StratifiedKFold(10, shuffle=True, random_state=42)
gs   = GridSearchCV(pipe, {"clf__C": [0.01, 0.1, 1, 10]},
                    scoring="roc_auc", cv=cv, n_jobs=-1)
gs.fit(df, y)
print(gs.best_score_, gs.best_params_)

# 5+6 Eval + deployment skeleton
import joblib, mlflow
joblib.dump(gs.best_estimator_, "readmit_v1.joblib")
mlflow.log_metric("auroc", gs.best_score_)
💡
提醒：在程式碼中為每個 CRISP-DM 階段加上明顯註解（如 # 3. Data Preparation），未來自己或他人接手時可以清楚追蹤流程，這是 reproducibility 的第一步。 Pro tip: Mark each CRISP-DM phase with an explicit comment (e.g. # 3. Data Preparation) in your code. When you or others revisit later, the workflow is traceable — this is reproducibility step #1.
