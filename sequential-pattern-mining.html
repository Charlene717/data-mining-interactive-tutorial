故事開場

場景：ICU 的 24 小時黃金窗口

SCENE · ICU 2018

敗血症發作前的 5 步指紋

Mayo Clinic 的 ICU 研究員 Dr. Chen 看著螢幕上 12,000 位過去 5 年的病人事件序列：抽血、生命徵象、用藥、報告⋯⋯她要找的是「敗血症 (sepsis) 發作前 24 小時的共通模式」。關聯規則探勘失敗了——它能告訴你「乳酸 + 心搏過速 + 體溫 + 血壓下降 + 白血球升高」共現，但不知道哪個先發生。在重症照護裡，順序就是一切。

她改用 PrefixSpan。輸入 12,000 個事件序列，min_support = 0.5。第一個結果讓她屏住呼吸：⟨乳酸 ≥ 2 → 心搏 > 100 → 體溫 > 38°C → MAP < 65 → SOFA ≥ 2⟩——支援度 78%，意思是 78% 的後續敗血症病人，在發作前 24 小時都依「這個順序」走過這 5 步。前期 4 步若觸發，平均距離正式診斷還有 11.3 小時——足夠介入。

她把這個 5-event sequence 包裝成早期預警分數 (EWS)，加入 ICU 的監測系統。一年後的回顧顯示：敗血症相關死亡率降低 14%、ICU 住院日減少平均 1.8 天。關聯規則告訴你「什麼一起發生」、序列模式告訴你「按什麼順序」——而順序，常常是時間窗口的關鍵。

Mayo Clinic ICU researcher Dr. Chen is staring at 12,000 patient event sequences from the past 5 years: blood draws, vitals, meds, reports. Her question: "Is there a common pattern in the 24 h before sepsis onset?" Association mining failed — it could tell her "lactate + tachycardia + fever + hypotension + leukocytosis" co-occur, but couldn't say which came first. In critical care, order is everything.

She switches to PrefixSpan. Inputs 12,000 event sequences with min_support = 0.5. The first result makes her hold her breath: ⟨lactate ≥ 2 → HR > 100 → temp > 38°C → MAP < 65 → SOFA ≥ 2⟩ — support 78%, meaning 78% of patients who later developed sepsis went through these 5 steps in this order in the prior 24 hours. The first 4 steps trigger an average of 11.3 hours before formal diagnosis — enough lead time to intervene.

She wraps the 5-event sequence into an Early Warning Score (EWS), integrates it into the ICU monitoring system. One year later: sepsis-related mortality down 14%, ICU LOS down 1.8 days on average. Association rules tell you what co-occurs; sequence patterns tell you the order — and order is often the key to the intervention window.

核心詞彙

12 個關鍵術語

Sequence database
序列資料庫
每個人都有一串「他做過什麼、按什麼順序」的清單。
A list of "what each person did, in what order".
D = {S_1, ..., S_N}，每 S_i = ⟨I_1, I_2, ...⟩ 是 timestamp 順序的 itemset 序列。Bio 例：病人事件、scRNA-seq 時間軸、單細胞發育軌跡。
D = {S_1, ..., S_N} where each S_i = ⟨I_1, I_2, ...⟩ is an ordered itemset sequence. Bio examples: patient events, time-stamped scRNA-seq, developmental trajectories.
Subsequence
子序列
原序列「跳著拿」幾個 itemset，順序不變——這就是子序列。
Pick some itemsets from the original sequence, keep their order — that's a subsequence.
P = ⟨P_1, ..., P_m⟩ 是 S = ⟨S_1, ..., S_n⟩ 的子序列，當存在 1 ≤ j_1 < ... < j_m ≤ n 使 P_k ⊆ S_{j_k}。
P = ⟨P_1, ..., P_m⟩ is a subsequence of S = ⟨S_1, ..., S_n⟩ iff there exist 1 ≤ j_1 < ... < j_m ≤ n with P_k ⊆ S_{j_k}.
Sequence support
序列支援度
這個小順序在資料庫裡幾位人「走過」？
How many people in the DB went through this little order?
supp(P) = |{S ∈ D : P ⊆ S as subsequence}| / |D|。注意計算的是「序列數」非「事件數」。
supp(P) = |{S ∈ D : P ⊆ S as subsequence}| / |D|. Counts sequences, not events.
GSP
GSP
序列版的 Apriori：逐層產生候選順序、過濾。
Apriori for sequences: generate-then-test, level by level.
Generalized Sequential Patterns (Srikant-Agrawal 1996)；候選由 (k-1)-pattern 自連產生，可加 sliding window / min-gap / max-gap 時序限制。
Generalized Sequential Patterns (Srikant-Agrawal 1996); candidates from self-join of (k-1)-patterns; supports sliding window / min-gap / max-gap temporal constraints.
SPADE
SPADE
每個 item 記它「在哪個序列、哪個位置」出現——交集就知道共現順序。
Each item lists "which sequence, which position" — intersect lists to find co-order.
Zaki 2001（Machine Learning 42）；垂直 id-list = {(sid, eid)}，supp 由 id-list 交集計算。比 GSP 快 1.5–3×。
Zaki 2001 (Machine Learning 42); vertical id-list = {(sid, eid)}, support via id-list intersection. 1.5–3× faster than GSP.
PrefixSpan
PrefixSpan
挑一個常見開頭，把每個序列「砍頭」只看後面，再遞迴。
Pick a frequent prefix, chop each sequence to "what comes after", recurse.
Pei et al. 2001（ICDE）；pattern-growth + projected databases，無候選產生。長序列效率最佳；FP-Growth 的序列對應版本。
Pei et al. 2001 (ICDE); pattern-growth + projected databases, no candidate generation. Best on long sequences; the sequence-data analogue of FP-Growth.
Projected database
投影資料庫
給一個前綴，把每個序列「砍掉開頭」只留後面那段——就是這個前綴的「世界」。
Given a prefix, chop each sequence to "what came after" — that's the prefix's "world".
D|⟨α⟩ = {S' : S' = S 砍 α 後的後綴}。PrefixSpan 在 D|⟨α⟩ 上遞迴找 ⟨α, x⟩ 的 frequent extension。記憶體會隨深度膨脹。
D|⟨α⟩ = {S' : S' = suffix of S after α}. PrefixSpan recurses on D|⟨α⟩ to find frequent extensions ⟨α, x⟩. Memory grows with depth.
Closed sequential pattern
閉合序列模式
沒有更長的「同樣常見」延伸——是該支援度的代表。
No longer extension with the same support — the representative at that level.
P closed iff ∀Q ⊃ P (with P subsequence of Q), supp(Q) < supp(P)。CloSpan (Yan-Han-Afshar 2003) 高效輸出。無資訊損失但大幅壓縮。
P closed iff ∀Q ⊃ P (with P as subsequence of Q), supp(Q) < supp(P). CloSpan (Yan-Han-Afshar 2003) outputs efficiently. Lossless yet drastically compressed.
Process mining
流程探勘
把事件記錄變成可執行的「流程圖」，有並行、有選擇、有循環。
Turn an event log into an executable workflow — with parallels, choices, and loops.
van der Aalst 2004 起；輸入 event log (case_id, activity, timestamp)，輸出 Petri net 或 BPMN。可做 discovery、conformance、enhancement。
Started by van der Aalst 2004; input is event log (case_id, activity, timestamp), output is a Petri net or BPMN. Supports discovery, conformance, enhancement.
Alpha algorithm
α-演算法
第一個從事件記錄推導流程圖的方法。
First algorithm to derive a workflow from event logs.
van der Aalst-Weijters-Maruster 2004 (IEEE TKDE)；以 footprint matrix（>, →, ∥, #）推導 Petri net 結構；對 noise 敏感。
van der Aalst-Weijters-Maruster 2004 (IEEE TKDE); derives Petri net structure from footprint matrix (>, →, ∥, #); noise-sensitive.
Inductive miner
Inductive Miner
用「分而治之」遞迴拆解事件記錄，保證輸出合理的流程圖。
Divide-and-conquer recursion that guarantees a sound workflow.
Leemans-Fahland-van der Aalst 2013 (CAiSE)；遞迴在 directly-follows graph 上找 cuts (XOR/AND/Loop)；soundness guaranteed；對 noise 穩健。
Leemans-Fahland-van der Aalst 2013 (CAiSE); recursively finds cuts (XOR/AND/Loop) in the directly-follows graph; soundness guaranteed; noise-robust.
Conformance checking
合規性檢查
每位病人實際的流程，跟「應該」的指引差多少？
How far is each patient's actual flow from the "should-be" guideline?
給定 event log + reference model，計算 fitness (能 replay 的比例) + precision (model 是否過泛)。常用 token-based replay 或 alignment-based。PM4Py 提供 API。
Given event log + reference model, compute fitness (fraction replayable) + precision (whether the model is too permissive). Common methods: token-based replay, alignment-based. PM4Py exposes the API.

PLAYLIST ANALOGY

序列模式 = Spotify 的「歌曲順序模式」

關聯規則告訴你「聽 A 的人也聽 B」（A 與 B 同籃子）；序列模式告訴你「聽 A 之後常接著聽 B、再接著聽 C」（順序、間隔皆重要）。
流程探勘則是把「歌曲序列」升級成「歌單樂譜」——不只是常見順序，還可以有「兩個動作可並行」、「這裡有循環」、「這裡只能擇一」的結構。生醫上：序列模式 = 病人事件常見順序；流程探勘 = 可執行的臨床路徑圖。

Association rules tell you "A-listeners also listen to B" (A and B share a basket); sequential patterns tell you "A is often followed by B, then C" (order and gaps matter).
Process mining upgrades the "song sequence" to a "music score" — beyond common orders, it captures "these two can happen in parallel", "here we loop", "here we pick one or the other". Biomedically: sequential patterns = common patient event orders; process mining = an executable clinical pathway graph.

點開看：closed vs maximal vs frequent 三層輸出有什麼差？
frequent：所有 supp ≥ minsup 的序列——最多、最完整、最冗。
closed：frequent 中「沒有更長的同 supp 超序列」——無損壓縮，從 closed 可還原所有 frequent 的 supp。
maximal：frequent 中「沒有任何 frequent 超序列」——最稀少，會丟失中間 supp 資訊。
比例（典型）：frequent : closed : maximal ≈ 100 : 30 : 5。實務上 closed 是最佳折衷——簡潔且資訊完整。
frequent: all sequences with supp ≥ minsup — most complete, most redundant.
closed: frequent with no longer same-supp super-sequence — lossless compression, all frequent supports recoverable from closed.
maximal: frequent with no frequent super-sequence — fewest, but loses intermediate supp info.
Typical ratio: frequent : closed : maximal ≈ 100 : 30 : 5. Closed is the sweet spot — concise yet information-complete.

概覽

序列 vs 項集

關聯規則的輸入是 {A, B, C} 的集合，順序無關。序列 (sequence) 則是 ⟨(A)(B, C)(D)⟩ 這樣的有序事件集合——每個括號是同一時刻的 itemset，順序代表時間。一個序列「支援」另一序列若後者是子序列（順序保持但可跳項）。

生物醫學上：clinical pathway（住院→檢查→診斷→處方→出院）、用藥順序（先 metformin → 加 insulin）、單細胞發育軌跡、ICD 共現的時序版本，全都是序列資料。

Association rules treat the input as an unordered set {A, B, C}. A sequence is an ordered list of itemsets like ⟨(A)(B, C)(D)⟩ — each bracket is an itemset at one timestamp, the order encodes time. One sequence "supports" another if the latter is a subsequence (order preserved, gaps allowed).

Biomedical examples: clinical pathways (admit → labs → diagnosis → meds → discharge), medication sequences (metformin first, insulin added later), single-cell developmental trajectories, time-ordered ICD code co-occurrences.

一、四大序列演算法

演算法策略特色 / 缺點
GSP (1996)逐層搜尋 + 時序限制直觀；候選爆炸；可加 sliding window / min-gap / max-gapIntuitive; candidate explosion; supports sliding window / min-gap / max-gap
SPADE (2001)垂直 id-list透過 id-list 交集計算支援度，效率比 GSP 高Support via id-list intersection — faster than GSP
PrefixSpan (2001)pattern-growth + projection不產候選；長序列上效率最佳；最常用No candidate generation; best on long sequences; most popular
CloSpan (2003)只保留 closed 序列輸出大幅壓縮、無資訊損失Vastly fewer patterns, lossless
BIDE另一 closed 演算法記憶體更省Lower memory than CloSpan
💡
PrefixSpan 的核心：取一個常見前綴（如 ⟨A⟩），把所有序列「投影」成 A 之後的尾巴（投影資料庫），再對投影資料庫遞迴找頻繁前綴——所以叫 pattern-growth。不用枚舉候選，直接成長。

 PrefixSpan's core: take a frequent prefix (e.g., ⟨A⟩), project every sequence to whatever comes after A (the projected DB), then recursively mine frequent prefixes there — hence pattern-growth. No candidates: patterns are grown directly.

流程探勘

二、流程探勘：序列的圖形化升級

序列模式探勘輸出「常見子序列」清單，流程探勘進一步輸出可執行的工作流程圖 (Petri net / BPMN)。輸入也是 event log（每筆 = case_id, activity, timestamp），但輸出有並行、選擇、循環等結構，更接近真實流程。

三大發現演算法：

  • Alpha-algorithm：從 event log 直接推導 Petri net；對雜訊敏感。
  • Inductive Miner：以分治遞迴拆解 log，保證輸出合理 (sound) 的流程；對雜訊較穩健。
  • Heuristic Miner：以依賴度閾值剪枝，可處理含噪 log。

常用工具：Python 的 pm4py、ProM、Disco、Celonis。生醫應用：急診流程瓶頸分析、術後恢復路徑、與 clinical guideline 的順從度檢驗 (conformance checking)。

Sequential mining outputs a list of frequent subsequences. Process mining goes further: it outputs an executable workflow graph (Petri net / BPMN). Input is an event log (case_id, activity, timestamp), but output has parallelism, choice, and loops — much closer to real-world workflows.

Three discovery algorithms:

  • Alpha-algorithm: derives Petri nets directly from event log; noise-sensitive.
  • Inductive Miner: divide-and-conquer recursion, guarantees a sound model; noise-robust.
  • Heuristic Miner: prunes by dependency threshold, handles noisy logs.

Common tools: Python pm4py, ProM, Disco, Celonis. Biomedical uses: ED throughput bottlenecks, post-op recovery pathways, conformance checking against clinical guidelines.

⚠️

逐步演算

PrefixSpan 手算：4 個序列

6

4 個序列

minsup=2 (50%)
Step 1
掃 DB 找 1-pattern。計算每個 item 的「sequence support」（含該 item 的序列數）：a: 3 (S1, S3, S4)、b: 3 (S1, S2, S3)、c: 4 (全部)、d: 2 (S2, S3)。
L1 = {a, b, c, d}（皆 ≥ 2 minsup）。 Scan DB for 1-patterns. Compute sequence support per item (number of sequences containing it): a: 3 (S1, S3, S4), b: 3 (S1, S2, S3), c: 4 (all), d: 2 (S2, S3).
L1 = {a, b, c, d} (all ≥ 2 minsup).
Step 2
對前綴 ⟨a⟩ 構建 projected DB。每個含 a 的序列「砍去 a 之前」只看後綴：D|⟨a⟩ = {⟨b,c⟩ (from S1), ⟨b,c,d⟩ (from S3), ⟨c⟩ (from S4)}。 Build projected DB for prefix ⟨a⟩. For each sequence containing a, take the suffix after a: D|⟨a⟩ = {⟨b,c⟩ (from S1), ⟨b,c,d⟩ (from S3), ⟨c⟩ (from S4)}.
Step 3
在 D|⟨a⟩ 中找頻繁 item。計算後綴內每個 item 的 sequence support：b: 2、c: 3、d: 1。b、c ≥ minsup，d 剪掉。延伸：⟨a, b⟩ supp = 2、⟨a, c⟩ supp = 3。 Find frequent items in D|⟨a⟩. Count sequence support of each item in the suffixes: b: 2, c: 3, d: 1. b, c pass; d pruned. Extend: ⟨a, b⟩ supp = 2, ⟨a, c⟩ supp = 3.
Step 4
遞迴：對 ⟨a, b⟩ 投影。D|⟨a,b⟩ = {⟨c⟩ (S1), ⟨c,d⟩ (S3)}。其中 c: 2, d: 1。⟨a, b, c⟩ supp = 2（保留）。再對 ⟨a, b, c⟩ 投影 → D|⟨a,b,c⟩ = {空, ⟨d⟩}, d: 1 < minsup，停止。 Recurse: project on ⟨a, b⟩. D|⟨a,b⟩ = {⟨c⟩ (S1), ⟨c,d⟩ (S3)}. c: 2, d: 1. ⟨a, b, c⟩ supp = 2 (keep). Project on ⟨a, b, c⟩ → D|⟨a,b,c⟩ = {empty, ⟨d⟩}, d: 1 < minsup, stop.
Step 5
同理對 ⟨b⟩、⟨c⟩、⟨d⟩。D|⟨b⟩ = {⟨c⟩, ⟨c,d⟩, ⟨c,d⟩}, c: 3, d: 2 → ⟨b, c⟩ supp=3、⟨b, d⟩ supp=2、⟨b, c, d⟩ supp=2。D|⟨c⟩ = {空, ⟨d⟩, ⟨d⟩, 空}, d: 2 → ⟨c, d⟩ supp=2。D|⟨d⟩ = 全空後綴，無延伸。 Same for ⟨b⟩, ⟨c⟩, ⟨d⟩. D|⟨b⟩ = {⟨c⟩, ⟨c,d⟩, ⟨c,d⟩}, c: 3, d: 2 → ⟨b, c⟩ supp=3, ⟨b, d⟩ supp=2, ⟨b, c, d⟩ supp=2. D|⟨c⟩ = {empty, ⟨d⟩, ⟨d⟩, empty}, d: 2 → ⟨c, d⟩ supp=2. D|⟨d⟩ has only empty suffixes, no extensions.
Step 6
最終頻繁序列模式。1-pattern: a, b, c, d；2-pattern: ⟨a,b⟩, ⟨a,c⟩, ⟨b,c⟩, ⟨b,d⟩, ⟨c,d⟩；3-pattern: ⟨a,b,c⟩, ⟨b,c,d⟩。共 11 個頻繁模式，全部支援度 ≥ 2。觀察：PrefixSpan 完全沒產候選——每一步都從投影 DB 直接「成長」，這是 pattern-growth 的核心優勢。 Final frequent sequences. 1-pattern: a, b, c, d; 2-pattern: ⟨a,b⟩, ⟨a,c⟩, ⟨b,c⟩, ⟨b,d⟩, ⟨c,d⟩; 3-pattern: ⟨a,b,c⟩, ⟨b,c,d⟩. 11 frequent patterns total, all support ≥ 2. Observation: PrefixSpan never generates candidates — every step grows directly from the projected DB. That's the core advantage of pattern-growth.
關鍵觀察：對應 GSP 會產生 16 個 candidate 2-pattern 並逐一掃 DB；PrefixSpan 直接從投影 DB 取頻繁延伸——長序列效率差 10× 以上。
思辨

三個常被爭辯的問題

Q · WHEN

序列 vs 項集——何時必須用序列？

關聯規則處理「同籃子」，序列處理「同籃子 + 順序」。何時需要順序？

Association rules handle "same basket"; sequences handle "same basket + order". When do you need order?

必須用序列： 「臨床診斷與處方的時序、購物路徑、網頁瀏覽軌跡、單細胞發育、犯罪事件先後。」 "Clinical dx-then-rx, shopping paths, clickstreams, single-cell development, crime event sequences."
不需要序列： 「購物籃內容（一次結帳的所有東西順序無關）、ICD 碼共現（同次住院的所有 code）、SNP 共現。」 "Shopping cart contents (order within one checkout irrelevant), ICD codes in one admission, SNP co-occurrence."
灰色地帶： 「同次門診的處方順序——若藥師按處方先後配藥，順序就重要；若一起發出，順序無意義。」 "Order of meds in one clinic visit — matters if dispensed sequentially, irrelevant if dispensed together."
Q · OUTPUT

閉合 vs 最大序列模式——用哪一個？

closed（沒有更長同 supp 延伸）保留所有支援度資訊；maximal（沒有任何 frequent 延伸）最精簡。實務上：(1) 展示用 maximal——只看「最完整的故事」；(2) 下游分析（如序列分類、序列預測）用 closed——資訊不損失；(3) 互動探索用 frequent + Top-N by support。

Closed (no longer same-supp extension) keeps all support info; maximal (no frequent extension) is leanest. In practice: (1) for display: maximal — only show "the fullest stories"; (2) for downstream (sequence classification, prediction): closed — no info loss; (3) for interactive exploration: frequent + Top-N by support.

Q · BOUNDARY

流程探勘是「序列探勘的延伸」還是「另一個領域」？

技術上有重疊但目標不同：序列探勘輸出「常見子序列清單」，流程探勘輸出「可執行的工作流圖（含並行、選擇、迴圈）」。前者來自 IBM Almaden（Agrawal 1995），後者來自 TU/e（van der Aalst 2004）的 BPM 社群。實務上常並用：序列探勘找熱門事件鏈、流程探勘建可審計的流程圖。在臨床上，PM4Py 已內建 conformance checking——比對病人實際路徑與臨床指引差異，常用於 quality improvement。

Technical overlap but different goals: sequence mining outputs "lists of frequent subsequences"; process mining outputs "executable workflow graphs (with parallelism, choice, loops)". The former from IBM Almaden (Agrawal 1995), the latter from TU/e's BPM community (van der Aalst 2004). They're often used together: sequence mining finds hot event chains; process mining builds auditable workflow maps. Clinically, PM4Py ships conformance checking — comparing actual patient pathways to guidelines, common in quality improvement.

🏥
SPOTLIGHT · 2018

Mayo Clinic 糖尿病足潰瘍治療路徑

Mayo Clinic 2018 年於 BMJ Open 發表的研究使用 PrefixSpan + Inductive Miner 分析 4,872 位糖尿病足潰瘍（DFU）患者的 5 年事件序列，包括清創、抗生素、影像、血管手術、最終截肢等 38 種 activity。發現：(1) 一條「延誤路徑」——抗生素未含 MRSA 涵蓋 → 3 週後復發 → 6 個月後截肢，占 23% 患者；(2) 一條「最佳路徑」——早期 vascular consult + 多學科團隊介入，截肢率 8% vs 全體 17%。將最佳路徑寫入 EHR 預設建議後，一年期截肢率降低 17%，估計每年挽救 1,200 例截肢。方法亮點：序列探勘列出常見事件鏈，流程探勘建構可比對的標準流程圖，conformance checking 量化每位病人的偏離。

Mayo Clinic's 2018 BMJ Open study used PrefixSpan + Inductive Miner on 4,872 diabetic foot ulcer (DFU) patients' 5-year event sequences, covering 38 activities (debridement, antibiotics, imaging, vascular surgery, eventual amputation). Findings: (1) a "delayed path" — antibiotics without MRSA coverage → recurrence at 3 weeks → amputation at 6 months, in 23% of patients; (2) a "best path" — early vascular consult + multidisciplinary team, amputation rate 8% vs 17% overall. After writing the best path into EHR default suggestions, 1-year amputation rate dropped 17%, estimated to save 1,200 amputations per year. Methodology highlight: sequence mining listed common event chains; process mining built a standard workflow for comparison; conformance checking quantified each patient's deviation.

陷阱

序列模式探勘 五大常見錯誤

序列當項集

把病人 5 年事件全當「一個 itemset」丟給 Apriori，失去順序。對策：確認「順序對你的問題重要嗎？」是，就用 PrefixSpan / SPADE。

Dumping 5 years of patient events as one itemset into Apriori discards order. Fix: ask "does order matter?" If yes, use PrefixSpan / SPADE.

minsup 太低

序列模式遠比項集多——10,000 個序列 × min_support=0.01 可能 1 億個 pattern。對策：從 0.3 起跑、視業務調整；考慮 closed/maximal 壓縮。

Sequential patterns explode faster than itemsets — 10K sequences × min_support=0.01 can blow to 100M patterns. Fix: start at 0.3, adjust; use closed/maximal compression.

時間間隔被忽略

「住院 → 截肢」這個序列若間隔 6 個月與間隔 6 年完全不同意義。對策：用 cSPADE / GSP 加 max-gap / min-gap / sliding window 約束。

"Admission → amputation" with a 6-month gap vs 6-year gap means utterly different things. Fix: use cSPADE / GSP with max-gap / min-gap / sliding-window constraints.

把整個 log 當一個序列

把所有病人的事件串成一條時間軸——你會得到全部支援度=1 但毫無意義的模式。對策：確保 case_id（病人 / 訂單）正確分割序列。

Concatenating every patient's events into one timeline yields all-support=1 nonsense. Fix: case_id (patient / order) must correctly split sequences.

Concept drift 不處理

2015 年的處方序列與 2025 年的差異很大（新藥、新指引）。把 10 年資料當靜態挖掘會混淆時代效應。對策：分時段挖掘、或用 streaming sequential mining。

2015 prescription sequences look nothing like 2025 (new drugs, new guidelines). Mining 10 years as static blurs era effects. Fix: mine by epoch, or use streaming sequential mining.

無 baseline 對照

「敗血症病人有 78% 走過這 5 步」——但非敗血症病人呢？也可能 60%。對策：對病例組與對照組分別挖掘，比較差異或計算 sequence-level lift。

"78% of sepsis patients went through these 5 steps" — but non-sepsis patients? Maybe 60%. Fix: mine cases vs controls separately; compare deltas or compute sequence-level lift.

不好的做法

把 10 年所有病人事件當成一個大序列，PrefixSpan min_support=0.001 跑 12 小時，輸出 5,000 萬個 pattern，按長度排序取 top-10 寫進論文。

Concatenate 10 years of all patient events as one big sequence, run PrefixSpan min_support=0.001 for 12 hours, output 50M patterns, sort by length, paper the top 10.

好的做法

(1) case_id = patient_id 分割；(2) 限定 5 年內事件 + max-gap 2 週；(3) PrefixSpan min_support=0.3，closed=True；(4) 與對照組（無敗血症）比 sequence-level lift；(5) 用 PM4Py inductive miner 建可比對的標準流程圖；(6) token-based replay 計算 fitness。

(1) case_id = patient_id; (2) restrict to ≤5 yr events + max-gap 2 weeks; (3) PrefixSpan min_support=0.3 with closed=True; (4) compare sequence-level lift vs non-sepsis controls; (5) build a standard workflow via PM4Py inductive miner; (6) compute fitness via token-based replay.

2024–2026 最前沿

SOTA · 2024–2026

從 PrefixSpan 到 LLM 與流式

LLM-augmented process mining（ICPM 2024）。傳統 Petri net 的 activity 名稱常是無意義代碼（如 "EVT_2941"）；2024 年起 GPT-4 / Claude 被用來：(1) 從 case description 自動命名 activity、(2) 解釋 Petri net 的業務含義、(3) 輔助 conformance checking 的 root cause（為何 case X 偏離？）。PM4Py 5.0 已加入 llm_assist module。

LLM-augmented process mining (ICPM 2024). Petri net activities often have meaningless codes ("EVT_2941"); since 2024, GPT-4 / Claude are used to: (1) auto-name activities from case descriptions, (2) explain Petri net business meaning, (3) assist conformance root-cause analysis ("why did case X deviate?"). PM4Py 5.0 added an llm_assist module.

流式序列模式探勘（IoT/ICU 監測 2024–2025）。傳統 PrefixSpan 是 batch，無法處理連續到達的事件流。Stream-PrefixSpan、Online-SPADE 引入 sliding-window + decay factor，使 minsup 隨時間自適應。應用於 ICU 即時敗血症監測——每分鐘更新「過去 24 小時內的高頻事件序列」。

Streaming sequential mining (IoT / ICU 2024–2025). Classic PrefixSpan is batch and can't handle continuous event streams. Stream-PrefixSpan, Online-SPADE add sliding-window + decay factor so minsup adapts over time. Applied in real-time ICU sepsis monitoring — every minute updates "high-frequency sequences in the past 24 h".

CRISP-DM for event logs（2025）。傳統 CRISP-DM 步驟對結構化資料設計；2025 ICPM 提出 PM²/CRISP-DM-for-event-logs，新增「process discovery」、「conformance」、「enhancement」三個階段，並強調 case_id 設計、time-window selection、log filtering。

CRISP-DM for event logs (2025). Classic CRISP-DM is designed for structured data; 2025 ICPM proposed PM²/CRISP-DM-for-event-logs adding "process discovery", "conformance", "enhancement" stages and emphasizing case_id design, time-window selection, log filtering.

序列分類 vs 序列模式探勘。序列分類（如 BERT、Transformer-on-events）以監督學習預測 outcome；序列模式探勘是無監督地找頻繁模式。兩者互補：先用序列模式探勘找 candidate features，再餵入 Transformer 做分類。2024 npj Digital Medicine 的 ICU 預警模型即此架構。

Sequence classification vs sequential pattern mining. Sequence classification (BERT, Transformer-on-events) is supervised outcome prediction; sequential pattern mining is unsupervised pattern discovery. Complementary: mine candidate features first, feed them to a Transformer classifier. The 2024 npj Digital Medicine ICU warning model uses exactly this stack.

He started the IEEE Task Force on Process Mining in 2009; his student Alessandro Berti released PM4Py in 2018; by 2024 the industry (Celonis is now valued at $13 B) has dwarfed the academic side. A spin-off of Apriori that grew into its own industry — that's the data-mining story.
