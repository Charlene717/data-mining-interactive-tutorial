CHAPTER 2 / 20 · MODULE A

資料準備與前處理

「Garbage in, garbage out」——再強的模型也無法救回失準的資料。

"Garbage in, garbage out" — no model can rescue ill-prepared data.

概覽

為什麼資料準備是最關鍵的階段？

資料準備（含清理、整合、變形、特徵工程）通常佔資料探勘專案 60–80% 的時間。它決定了模型輸入的「天花板」——若資料含大量缺失、編碼不一致、外洩 (target leakage)，再強的演算法也只會放大雜訊。

在生物醫學脈絡下，這個階段更具挑戰：EHR 缺失非隨機 (MNAR)、不同醫院 ICD 碼版本不一、檢驗值單位混雜 (mg/dL vs mmol/L)、組學批次效應 (batch effect)。許多失敗的探勘專案——例如多個 sepsis 早期預警模型於外院驗證崩潰——根因都出在這一步。

Data Preparation (cleaning, integration, transformation, feature engineering) typically consumes 60–80% of the project time. It defines the ceiling of model input — if the data has heavy missingness, inconsistent encoding, or target leakage, the strongest algorithm only amplifies noise.

In a biomedical context, this phase is even harder: EHR missingness is non-random (MNAR), ICD versions differ across hospitals, lab units are mixed (mg/dL vs mmol/L), and omics batch effects dominate signal. Many failed mining projects — e.g., several sepsis early-warning models collapsing at external validation — trace back to this step.

🚨
Target leakage 警告：「為什麼我的 AUROC = 0.99？」十次有九次的答案是target leakage：例如把出院後的 discharge_diagnosis 餵進 readmit 模型；或把 follow-up CRP 當作 sepsis 預測特徵。任何使用「未來才會知道」的資訊都是不可接受的。 Target leakage warning: "Why is my AUROC = 0.99?" Nine times out of ten the answer is target leakage — e.g., feeding post-discharge diagnosis into a readmission model, or treating follow-up CRP as a sepsis predictor. Any feature that uses "future" information is unacceptable.
核心步驟

一、六個關鍵子步驟

🧼

① 清理 Cleaning

處理缺失值、重複列、不一致拼寫（如 'M' vs 'Male'）、單位轉換、時間戳對齊。

Handle missing values, duplicate rows, inconsistent spellings ('M' vs 'Male'), unit conversion, timestamp alignment.

🔗

② 整合 Integration

串接多個來源（demographic + lab + medication），處理鍵衝突、版本歧異與時間視窗對齊。

Join multiple sources (demographic + lab + medication); resolve key conflicts, version drift, time-window alignment.

✂️

③ 切割 Splitting

Train / Validation / Test split；時序資料用時間切割，醫院資料用 by-hospital 切割避免 leakage。

Train / Validation / Test split; use time-based split for temporal data, by-hospital split to avoid leakage.

⚖️

④ 縮放 Scaling

Min-Max、Z-score、Robust、Quantile。距離型 (KNN/SVM/K-Means) 必須；樹型則不需要。

Min-Max, Z-score, Robust, Quantile. Mandatory for distance-based (KNN/SVM/K-Means); not needed for tree-based.

🔢

⑤ 編碼 Encoding

One-Hot、Ordinal、Target、Frequency；高基數類別建議 Target Encoding + CV-fold。

One-Hot, Ordinal, Target, Frequency. For high-cardinality use Target Encoding + CV-fold guarding.

⚙️

⑥ 特徵工程

分箱、log/exp、交互特徵、時間特徵（season、weekday）、領域知識特徵（BMI、CHA₂DS₂-VASc）。

Binning, log/exp, interaction terms, time features (season, weekday), domain features (BMI, CHA₂DS₂-VASc).

互動模擬

不平衡資料與重採樣效果

調整類別比例，觀察 SMOTE / Random Undersampling / Class Weight 對分類模型的影響。實務上，敗血症 ~5%、罕病 <1% 都是常見的不平衡比例。

Adjust class ratio and observe SMOTE / Random Undersampling / Class Weight effects. In practice sepsis ~5%, rare disease <1% are typical imbalances.

缺失值策略

二、缺失值類型與處理

類型意義處理建議
MCAR完全隨機缺失，與任何變數無關listwise deletion、中位數補Missing completely at randomListwise deletion, median imputation
MAR隨機，但與其他觀測變數有關MICE、KNN imputer、迴歸補Missing at random, depends on observedMICE, KNN imputer, regression-based
MNAR非隨機（如 viral load 太高未測即死亡）須建模化原因；用 missingness indicatorNot at random (e.g., viral load not measured before death)Model the cause; use missingness indicator

🌳 缺失值處理決策樹

Q1:
缺失率 < 5% 且樣本量充足？ → 是 → 直接 listwise delete。
Q2:
是 MCAR/MAR？ → 是 → MICE（多重補值，建議 m=5–20 次）。
Q3:
樹型模型如 XGBoost？ → 是 → 直接交給 model 處理 NA（sklearn 1.6 後 ExtraTrees 也支援）。
Q4:
懷疑 MNAR？ → 是 → 加 is_missing 二元指標當特徵，避免直接補導致偏差。
Q1:
Missingness < 5% with abundant samples? → Yes → Listwise delete.
Q2:
MCAR/MAR? → Yes → MICE (multiple imputation, m=5–20).
Q3:
Tree-based model (XGBoost)? → Yes → Let the model handle NA natively (sklearn 1.6+ also for ExtraTrees).
Q4:
Suspect MNAR? → Yes → Add an is_missing indicator as feature; never directly impute.
程式碼

實作：Pipeline + Imputer + Scaler

library(tidymodels); library(mice); library(themis)

# 1) 缺失值多重補值 / Multiple imputation
imp <- mice(df, m = 5, method = "pmm", seed = 42)
df_imp <- complete(imp, 1)

# 2) recipe：縮放 + 編碼 + SMOTE / scale + encode + SMOTE
rec <- recipe(target ~ ., data = df_imp) |>
  step_impute_knn(all_predictors(), neighbors = 5) |>
  step_normalize(all_numeric_predictors()) |>
  step_dummy(all_nominal_predictors()) |>
  step_smote(target, over_ratio = 1.0)         # themis

# 3) Train/test 切割避免 leakage
sp <- initial_split(df_imp, prop = 0.8, strata = target)
prep <- prep(rec, training = training(sp))
tr <- bake(prep, training(sp))
te <- bake(prep, testing(sp))   # test 不可再 SMOTE
from sklearn.compose       import ColumnTransformer
from sklearn.impute        import SimpleImputer, KNNImputer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from imblearn.over_sampling import SMOTE
from imblearn.pipeline      import Pipeline   # note imblearn pipeline
from sklearn.model_selection import train_test_split

X_tr, X_te, y_tr, y_te = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42)

pre = ColumnTransformer([
    ("num", Pipeline([
        ("imp",   KNNImputer(n_neighbors=5)),
        ("sc",    StandardScaler())]), num_cols),
    ("cat", OneHotEncoder(handle_unknown="ignore"), cat_cols)
])

pipe = Pipeline([
    ("pre",   pre),
    ("smote", SMOTE(random_state=42)),     # test 不會被 SMOTE
    ("clf",   XGBClassifier(n_estimators=300))
])
pipe.fit(X_tr, y_tr)
⚠️
SMOTE 的常見錯誤：絕對不能在 train/test split 之前 做 SMOTE！否則合成樣本會洩漏到測試集。Pipeline 內的 imblearn.Pipeline 會自動只對 train fold 套用。 SMOTE pitfall: Never apply SMOTE before train/test split — synthesized samples will leak into the test fold. The imblearn Pipeline automatically applies SMOTE only on training folds.
